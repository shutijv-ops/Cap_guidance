🔌 API Endpoints by Category

Authentication & Session Management

Method Endpoint Description
POST /api/admin/login Admin/Counselor Login
GET /api/admin/check Verify Session Status
POST /api/admin/logout Logout & Clear Session

Appointment Management

Method Endpoint Description
POST /api/appointments/student Student Submit Appointment
POST /api/appointments/request Request New Appointment
POST /api/appointments Create Appointment (Admin)
GET /api/appointments List All Appointments
GET /api/appointment/:ref Get Appointment Details
PUT /api/appointments/:ref Update Appointment Status

Counselor Management

Method Endpoint Description
GET /api/counselors List All Counselors
POST /api/counselors Create New Counselor
PUT /api/counselors/:id Update Counselor Profile
GET /api/counselor Get Current Counselor Info

Schedule & Notifications

Method Endpoint Description
GET /api/schedules/:date Get Schedules for Date
GET /api/bookedSlots Get All Booked Time Slots
GET /api/notifications/sse Server-Sent Events Stream
GET /api/notifications Get Notifications List
DELETE /api/notifications/:id Delete Notification

Reports & Utilities

Method Endpoint Description
GET /api/reports/monthly Monthly Statistics Report
POST /api/migrateApprovedSchedules Migrate Schedules
GET /health Health Check

📊 Appointment Lifecycle Data Flow

STEP 1: STUDENT INITIATES REQUEST ───────────────────────────────── Student Landing Page → View Counselors → Check Availability → Submit Appointment Request ↓ STEP 2: SERVER PROCESSING ───────────────────────── POST /api/appointments/student ├─ Validate Input ├─ Generate Reference Number ├─ Store in MongoDB └─ Trigger Notification ↓ STEP 3: REAL-TIME NOTIFICATION ────────────────────────────── GET /api/notifications/sse → Broadcast to Admin/Counselor Portal → "New Appointment Request!" Alert ↓ STEP 4: SEND CONFIRMATION EMAIL ─────────────────────────────── SendGrid API → Student receives email with Reference Number & Details ↓ STEP 5: ADMIN/COUNSELOR REVIEWS ─────────────────────────────── Admin Portal → GET /api/appointments → View Pending Requests → Select Appointment → Review Details ↓ STEP 6: MAKE DECISION ──────────────────── PUT /api/appointments/:ref ├─ Action: Approve / Reject / Reschedule ├─ Update Status in MongoDB ├─ Broadcast SSE Notification └─ Send Decision Email via SendGrid ↓ STEP 7: STUDENT RECEIVES UPDATE ─────────────────────────────── Email arrives → Student logs in → GET /api/appointments → Views Updated Status → Calendar reflects new appointment if approved ↓ STEP 8: ATTENDANCE & COMPLETION ─────────────────────────────── Appointment Date → Counselor Session → Notes recorded → Status marked complete → Database updated with session details ↓ STEP 9: REPORTS & ANALYTICS ─────────────────────────── GET /api/reports/monthly → System generates statistics → Admin views reports & analytics

💾 Database Schema

Counselor Collection

{ _id: ObjectId, title: String, firstName: String, middleName: String, lastName: String, email: String, username: String, password: String (hashed), role: String (Counselor/Admin), status: String (Active/Inactive), availability: Array, createdAt: Date, updatedAt: Date }

Appointment Collection

{ _id: ObjectId, refNumber: String (unique), studentId: String, studentName: String, email: String, course: String, year: String, date: Date, time: String, duration: Number, counselorId: ObjectId, counselorName: String, status: String (Pending/Approved/Rejected/Rescheduled), reason: String, notes: String, remarks: String, createdAt: Date, approvedAt: Date, updatedAt: Date }

Notification Collection

{ _id: ObjectId, message: String, type: String (Info/Alert/Success/Error), userId: String, appointmentRef: String, read: Boolean, createdAt: Date, updatedAt: Date }
